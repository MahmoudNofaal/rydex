# Rydex — Requirements and Risk

> [!abstract] What this file is This is the consolidated "what exactly, and what could go wrong" of Rydex — every feature's acceptance criteria, every NFR turned into a measurable SLO, and every risk on the project with an owner and a mitigation. It merges three previously separate documents (Requirements Traceability Matrix, SLO Definitions, Risk Register) into a single file. This is the file to open when planning a milestone, writing a test, or checking whether something is actually, verifiably done.

## Part G — Requirements Traceability Matrix

> [!abstract] Purpose of this document Per [[3-End-to-End-Delivery-Plan|Delivery Plan]] Phase 0.1, this file takes every row in [[1-PRD|PRD]] §6 (Feature Set) and maps it to: (1) acceptance criteria in Given/When/Then format, (2) a confirmed priority tag, and (3) which Epics block which. This is the file that turns the PRD from "what we want" into "what we're building," with no ambiguity left for engineering to guess at — per the Phase 0 exit criteria.

> [!important] Exit criteria for this file Per the Delivery Plan: "Every Epic in the PRD has acceptance criteria, every NFR has a number, and the team can explain _why_ each Epic exists without re-reading the PRD." This file satisfies the acceptance-criteria portion of that bar. NFR numbers are [SLO Definitions (Part H above)](https://claude.ai/chat/931cc535-a10d-4882-adff-50ece26a8b63#part-h--slo-definitions)'s job, not this file's.

> [!note] This file also resolves open questions raised earlier [[1.rydex-product-overview|feature roadmap]] §10 and [[1.rydex-product-overview|personas]] §3 flagged three unresolved items. This matrix is where they get explicitly closed (not silently — each resolution is marked inline):

> 2. Demand forecasting per zone milestone placement

## G1. How to Read This Matrix

Each feature carries:

- **Priority (confirmed)** — the PRD's original tag, validated against what the rest of this document set has established since; a ✅ means confirmed as-is, a 🔁 means changed with reasoning given
- **Acceptance criteria** — Given/When/Then, written at a level a QA engineer could turn directly into a test, per [[3-End-to-End-Delivery-Plan|Delivery Plan]] §3.6's testing philosophy
- **Blocks / blocked by** — which other Epics must exist first; this is dependency information the [[2-Engineering-Domain-Roadmap|Engineering Domain Roadmap]] and [[3-End-to-End-Delivery-Plan|Delivery Plan]] phase sequencing already assume but don't enumerate feature-by-feature

## G2. Epic A — Identity & Accounts

> [!quote] Blocks: every other Epic. Nothing in Rydex executes without an authenticated, role-aware actor.

### Rider signup/login

**Priority:** MVP ✅

**Acceptance criteria:**

- **Given** a person provides a valid email or phone number and password, **when** they submit signup, **then** an account is created and they are authenticated
- **Given** an existing rider provides correct credentials, **when** they submit login, **then** they receive a valid session/token scoped to the Rider role
- **Given** a person provides an email/phone already registered, **when** they attempt signup, **then** the system rejects it with a clear, non-leaking error (does not confirm which field is taken, per [[1-PRD|PRD]] §11 security framing)

**Blocked by:** None — this is a foundation feature. **Blocks:** Every Rider-facing feature in every other Epic.

### Driver onboarding

**Priority:** MVP ✅

**Acceptance criteria:**

- **Given** a person completes profile, vehicle info, and document upload, **when** they submit onboarding, **then** their account is created in a "pending verification" state, not yet eligible to go online
- **Given** a driver's documents are in "simulated verification," **when** an Admin reviews and approves them, **then** the driver's status changes to eligible-to-go-online
- **Given** a driver has not completed onboarding, **when** they attempt to go online, **then** the system blocks it with a clear reason

**Blocked by:** None — foundation feature, though it depends on Driver management (Epic I) existing for the approval step to have somewhere to go. **Blocks:** All Driver-facing features in Epic C, D, E.

> [!note] Cross-Epic coupling flagged Driver onboarding's "simulated verification" step is functionally incomplete without Epic I's Driver management (approve/suspend). These two features should be treated as a single deliverable for sequencing purposes even though they're in different Epics — see [[1.rydex-product-overview|feature roadmap]] §3, where both already land in Alpha together.

### Role-based access

**Priority:** MVP ✅

**Acceptance criteria:**

- **Given** a user is authenticated, **when** they access any endpoint or view, **then** they see only the views and data permitted for their role (Rider, Driver, Support Agent, Admin)
- **Given** a Rider attempts to access another rider's trip data, **when** the request is made, **then** it is rejected regardless of role correctness, because this is a resource-level check, not just a role-level one (per [[2-Engineering-Domain-Roadmap|Engineering Domain Roadmap]] 2.4 — Resource-based authorization)

**Blocked by:** Rider signup/login, Driver onboarding (need actors with roles to enforce against). **Blocks:** Every feature in every Epic — this is the authorization substrate the entire system depends on.

### Profile & preferences

**Priority:** MVP ✅

**Acceptance criteria:**

- **Given** a Rider or Driver is authenticated, **when** they update payment method, saved locations, or ride preferences, **then** the change is persisted and reflected on their next relevant action (e.g., next trip request pre-fills saved location)

**Blocked by:** Rider signup/login, Driver onboarding. **Blocks:** Payment capture (Epic E) — a saved payment method must exist before a trip can be charged.

## G3. Epic B — Trip Lifecycle

> [!quote] Blocks: Epic D (matching needs a trip to match), Epic E (pricing needs a trip to price), Epic F (fraud detection needs trip events to analyze), Epic C (status push needs trip status to exist first).

### Request a trip

**Priority:** MVP ✅

**Acceptance criteria:**

- **Given** an authenticated Rider sets a pickup and destination, **when** they submit a trip request, **then** a Trip record is created in `Requested` status and the matching process begins
- **Given** a Rider submits a duplicate request using the same Idempotency-Key (per [[2-Engineering-Domain-Roadmap|Engineering Domain Roadmap]] 2.5), **when** the retry is received, **then** no second Trip record is created

**Blocked by:** Rider signup/login, Role-based access (Epic A). **Blocks:** Nearest-driver matching (Epic D), Base fare calculation (Epic E).

### Trip status tracking

**Priority:** MVP ✅

**Acceptance criteria:**

- **Given** a Trip exists, **when** its state changes, **then** it transitions only through the legal sequence: `Requested → Matched → DriverEnRoute → InProgress → Completed | Cancelled`
- **Given** a Trip is in a terminal state (`Completed` or `Cancelled`), **when** any further transition is attempted, **then** it is rejected

**Blocked by:** Request a trip. **Blocks:** Live trip status push (Epic C), Trip history & receipts.

### Cancellation handling

**Priority:** MVP ✅

**Acceptance criteria:**

- **Given** a Trip is within its cancellation window, **when** a Rider or Driver cancels, **then** the Trip moves to `Cancelled`, no fee is applied, and a cancellation reason is captured
- **Given** a Trip is outside its cancellation window, **when** a cancellation is requested, **then** a fee is applied per pricing rules and the reason is still captured

**Blocked by:** Trip status tracking. **Blocks:** Cancellation rate metric (per [[1.rydex-product-overview|success metrics]] §4) — this feature is the only source of the reason data that makes that metric diagnostic rather than a bare count.

### Trip history & receipts

**Priority:** MVP ✅

**Acceptance criteria:**

- **Given** a Rider or Driver has completed or cancelled trips, **when** they request trip history, **then** they receive a paginated list (cursor-based, per [[2-Engineering-Domain-Roadmap|Engineering Domain Roadmap]] 2.5) showing only their own trips
- **Given** a completed Trip, **when** a receipt is requested, **then** it shows a full fare breakdown and is downloadable

**Blocked by:** Trip status tracking, Base fare calculation (Epic E). **Blocks:** None downstream.

### Ratings & feedback

**Priority:** Advanced ✅ — **🔁 Milestone placement now confirmed**

**Resolution of open question #1:** [[1.rydex-product-overview|feature roadmap]] §5 placed this in Beta 2 by inference, flagging it for confirmation here. **Confirmed: Beta 2 is correct.** Reasoning: the feature requires a completed trip lifecycle (present since Alpha) and is tagged Advanced, not MVP, so it has no reason to ship before Alpha closes — but it also has no dependency on Beta 1's real-time layer or Beta 2's matching/pricing intelligence, meaning it could technically ship as early as immediately-post-Alpha. **It is grouped into Beta 2 as a matter of release packaging convenience, not technical necessity.** This distinction matters for scheduling flexibility — if Beta 2 risks slipping, Ratings & feedback is a safe candidate to ship early rather than cut.

**Acceptance criteria:**

- **Given** a Trip reaches `Completed`, **when** either party submits a rating, **then** it is recorded and visible to platform-internal trust calculations but not to the other party until they've also rated (avoiding retaliatory rating bias)

**Blocked by:** Trip status tracking (must reach `Completed`). **Blocks:** None downstream, though it likely feeds fraud detection's signal set (Epic F) per [[2-Engineering-Domain-Roadmap|Engineering Domain Roadmap]] 3.5's "driver rating history" feature signal.

## G4. Epic C — Live Tracking & Real-Time Communication

> [!quote] Blocks: Epic D's migration to Redis geo-index (per [[3-End-to-End-Delivery-Plan|Delivery Plan]] Phase 4.3, live location must exist before matching can move off SQL).

### Live driver location on map

**Priority:** MVP ✅

**Acceptance criteria:**

- **Given** a Driver is online and has an active or pending trip, **when** their location updates, **then** the Rider's client receives the new position within the target real-time latency window (number defined in [SLO Definitions (Part H above)](https://claude.ai/chat/931cc535-a10d-4882-adff-50ece26a8b63#part-h--slo-definitions))
- **Given** a Driver goes offline mid-update-stream, **when** the disconnect is detected, **then** the Rider's client is informed rather than silently showing a stale position

**Blocked by:** Driver onboarding (Epic A), Trip status tracking (Epic B) — needs an active trip context to know which Rider should receive which Driver's location. **Blocks:** Migration of matching to Redis geo-index (Epic D, Beta 2).

### Live trip status push

**Priority:** MVP ✅

**Acceptance criteria:**

- **Given** a Trip's status changes, **when** the domain event fires, **then** both Rider and Driver clients receive the update without polling or refreshing
- **Given** a backend instance handling a connection fails, **when** failover occurs, **then** the status message is not lost (per [[2-Engineering-Domain-Roadmap|Engineering Domain Roadmap]] 3.1's explicit "must-not-lose" reliability tier for status, distinct from "lossy" location pings)

**Blocked by:** Trip status tracking (Epic B). **Blocks:** In-app trip notifications (this Epic).

### Driver presence (online/offline/busy)

**Priority:** Advanced ✅

**Acceptance criteria:**

- **Given** a Driver toggles online, **when** the toggle completes, **then** their status is `Online` and they become eligible for matching
- **Given** a Driver is mid-trip, **when** queried, **then** their presence shows `Busy`, making them ineligible for new offers even if technically connected

**Blocked by:** Driver onboarding (Epic A), Live driver location on map. **Blocks:** Fairness-aware queueing (Epic D) — the fairness queue's "became available" timestamp is meaningless without accurate presence state.

### In-app trip notifications

**Priority:** Advanced ✅

**Acceptance criteria:**

- **Given** a Driver arrives at the pickup location, **when** proximity is detected, **then** the Rider receives a "your driver has arrived" notification within the real-time latency target
- **Given** a Driver is approaching but not yet arrived, **when** a proximity threshold (e.g., 2 minutes out) is crossed, **then** the Rider receives a corresponding notification, sent once, not repeatedly

**Blocked by:** Live trip status push, Live driver location on map. **Blocks:** None downstream.

## G5. Epic D — Matching & Dispatch Intelligence

> [!quote] Per the Delivery Plan's own example: "Epic D batch matching needs Epic B trip lifecycle to exist first." This section confirms and extends that example across all four features.

### Nearest-driver matching

**Priority:** MVP ✅

**Acceptance criteria:**

- **Given** a Trip is `Requested` and available Drivers exist within search radius, **when** matching runs, **then** the closest available Driver receives an offer
- **Given** the offered Driver does not respond within the timeout window, **when** the timeout elapses, **then** the system automatically offers the next-best Driver

**Blocked by:** Request a trip (Epic B), Driver onboarding + presence (Epic A/C — at Alpha, presence is simplified since Redis geo-index doesn't exist yet, per [[3-End-to-End-Delivery-Plan|Delivery Plan]] §3.4). **Blocks:** Batch matching, Fairness-aware queueing (same Epic — these are alternative/successor strategies via the `IMatchingStrategy` interface).

### Batch matching

**Priority:** Advanced ✅

**Acceptance criteria:**

- **Given** multiple Trips are `Requested` within the same matching window, **when** batch matching runs, **then** the assignment minimizes aggregate pickup distance across the group, measurably better than running nearest-driver matching on each request individually
- **Given** a batch match completes, **when** compared against a naive nearest-match baseline on the same input, **then** the comparison is logged and reportable (this _is_ the Match performance benchmarking feature's data source)

**Blocked by:** Request a trip (Epic B) — confirming the Delivery Plan's own stated example. Also blocked by Live driver location on map (Epic C) and the Redis geo-index migration, since batch matching at scale needs fast candidate lookup. **Blocks:** Match performance benchmarking.

### Fairness-aware queueing

**Priority:** Advanced ✅

**Acceptance criteria:**

- **Given** two Drivers are equally good candidates by distance, **when** an offer is generated, **then** the Driver who has been waiting longer (per their "became available" timestamp) receives priority
- **Given** a Driver completes a trip and becomes available again, **when** their presence updates, **then** their fairness-queue timestamp resets to the current time, placing them at the back of the priority queue

**Blocked by:** Driver presence (Epic C) — the timestamp this feature depends on doesn't exist without presence tracking. **Blocks:** None downstream within this Epic, though it's a required input to Match performance benchmarking's fairness comparison.

### Match performance benchmarking

**Priority:** Advanced ✅

**Acceptance criteria:**

- **Given** Nearest-driver, Batch, and Fairness-aware strategies have all run against comparable inputs, **when** a benchmarking report is generated, **then** it reports both speed and fairness outcomes per strategy, not speed alone
- **Given** a new matching strategy is added in the future, **when** it implements `IMatchingStrategy`, **then** it can be benchmarked without modifying the dispatch service (per [[2-Engineering-Domain-Roadmap|Engineering Domain Roadmap]] 1.3's design intent)

**Blocked by:** Batch matching, Fairness-aware queueing (needs at least two strategies to compare). **Blocks:** None — this is a terminal/evidence feature, similar in role to the Release Candidate milestone's evidence-pass features.

## G6. Epic E — Pricing & Payments

> [!quote] Blocks: Trip history & receipts (Epic B, needs a fare to display), Driver payouts (needs base fare + surge to calculate from).

### Base fare calculation

**Priority:** MVP ✅

**Acceptance criteria:**

- **Given** a Trip's pickup and destination, **when** fare is calculated, **then** it reflects distance and time per the defined rate, with the calculation method deterministic and auditable

**Blocked by:** Request a trip (Epic B) — needs pickup/destination to exist. **Blocks:** Trip history & receipts (Epic B), Payment capture, Dynamic (surge) pricing (surge multiplies against this base).

### Dynamic (surge) pricing

**Priority:** Advanced ✅

**Acceptance criteria:**

- **Given** ride requests in a zone outpace available drivers over the rolling window, **when** a new fare is calculated in that zone, **then** a surge multiplier is applied and visible to the Rider before trip confirmation
- **Given** supply and demand normalize in a zone, **when** the rolling window reflects this, **then** the surge multiplier automatically removes itself without manual intervention

**Blocked by:** Base fare calculation, Zone & pricing rule management (Epic I — zones must be defined before per-zone surge can apply). **Blocks:** Driver payouts (surge bonus calculation).

### Payment capture

**Priority:** MVP ✅

**Acceptance criteria:**

- **Given** a Trip reaches `Completed`, **when** the final fare is calculated, **then** the Rider's saved payment method is charged automatically without requiring manual confirmation
- **Given** a payment attempt fails, **when** the failure is detected, **then** the Trip is not silently marked as paid, and the failure is surfaced for resolution rather than lost

**Blocked by:** Base fare calculation, Profile & preferences (Epic A — needs a saved payment method to exist). **Blocks:** Trip history & receipts (a receipt isn't meaningful without confirmed payment).

### Driver payouts

**Priority:** Advanced ✅

**Acceptance criteria:**

- **Given** a Trip is `Completed` and paid, **when** payout is calculated, **then** it reflects base fare share plus any surge bonus earned, and is visible to the Driver
- **Given** a surge bonus applied to a Trip, **when** the Driver views earnings, **then** the surge portion is itemized separately from the base payout, not blended into one number (this is what makes the metric in [[1.rydex-product-overview|success metrics]] §2's "fair, transparent" framing concrete)

**Blocked by:** Payment capture, Dynamic (surge) pricing. **Blocks:** None downstream.

## G7. Epic F — Trust & Safety

> [!quote] Blocks: nothing structurally, but is itself blocked by nearly everything — fraud detection has no signal to analyze until real trips, with real location streams, are flowing.

### Fraud/anomaly detection

**Priority:** Advanced ✅

**Acceptance criteria:**

- **Given** a Trip's location stream shows a position jump inconsistent with elapsed time and road-feasible speed, **when** the check runs, **then** the Trip is flagged with a risk score and routed to an Ops Agent review queue
- **Given** a Trip is flagged, **when** no human has reviewed it, **then** the system takes no automatic punitive action — flagging never auto-cancels (per [[1-PRD|PRD]] §10's explicit constraint)

**Blocked by:** Live driver location on map (Epic C) — needs a real location stream to analyze. Per [[3-End-to-End-Delivery-Plan|Delivery Plan]] §6.1, this consumes the location stream asynchronously, never on the critical matching path. **Blocks:** Dispute flagging (a flagged trip is a natural input to a dispute review, though the two can also originate independently).

### SOS / emergency flag

**Priority:** Stretch ✅

**Acceptance criteria:**

- **Given** a Rider or Driver is on an in-progress Trip, **when** they trigger SOS, **then** the Trip is immediately routed to an Ops Agent for urgent review, bypassing the normal flagged-trip queue priority
- **Given** an SOS is triggered, **when** the triggering party's app confirms it, **then** they receive confirmation the alert was received, so they're not left uncertain whether it worked

**Blocked by:** Trip status tracking (Epic B) — needs an in-progress trip context. **Blocks:** None downstream.

### Dispute flagging

**Priority:** Advanced ✅

**Acceptance criteria:**

- **Given** a Trip is `Completed`, **when** either Rider or Driver flags it for dispute, **then** it enters a review queue visible to Ops Agents with full trip context attached
- **Given** a disputed Trip, **when** an Ops Agent resolves it, **then** the resolution (and any refund/adjustment) is recorded against the Trip permanently, not just communicated verbally

**Blocked by:** Trip status tracking (Epic B). **Blocks:** Dispute resolution workflow (Epic G) — that feature is the structured version of what this feature creates raw entries for.

## G8. Epic G — AI-Assisted Support

> [!quote] Blocks: nothing structurally downstream, but depends on the platform's own documented policy content existing — a prerequisite the PRD assumes but doesn't assign to any Epic.

### AI support assistant

**Priority:** Advanced ✅

**Acceptance criteria:**

- **Given** a Rider or Driver asks a question covered by documented platform policy, **when** the assistant retrieves and answers, **then** the answer is grounded in that retrieved content, not general knowledge
- **Given** a question is outside what's documented, **when** the assistant has low retrieval confidence, **then** it says so explicitly rather than guessing (per [[1-PRD|PRD]] §10's "not a general chatbot" requirement)

**Blocked by:** None functionally, but practically requires the platform's policy documentation to exist as ingestable content — **this is a prerequisite not currently assigned to any Epic or owner; flagged below in §10.** **Blocks:** Smart escalation (this Epic).

### Smart escalation

**Priority:** Advanced ✅

**Acceptance criteria:**

- **Given** the AI support assistant cannot confidently answer, **when** it escalates, **then** a human Ops Agent receives the full prior conversation context and the Rider/Driver is not asked to repeat themselves
- **Given** an escalation occurs, **when** measured, **then** it contributes to the AI support deflection rate metric as a "not deflected" case, not silently dropped from the count

**Blocked by:** AI support assistant. **Blocks:** Dispute resolution workflow (for cases that escalate specifically around fare/refund disputes).

### Dispute resolution workflow

**Priority:** Stretch ✅

**Acceptance criteria:**

- **Given** a dispute has been flagged (Epic F) or escalated (Smart escalation, this Epic), **when** an Ops Agent processes it through the structured workflow, **then** the outcome (refund, adjustment, no action) is recorded with a reason
- **Given** a refund is issued through this workflow, **when** processed, **then** it correctly reverses the relevant portion of Payment capture (Epic E) rather than being tracked as a separate, disconnected ledger entry

**Blocked by:** Dispute flagging (Epic F), Smart escalation, Payment capture (Epic E). **Blocks:** None downstream — this is a terminal workflow feature.

## G9. Epic H — Analytics & Forecasting

> [!quote] Blocks: nothing — this Epic exists entirely downstream of data produced elsewhere; it has no features anything else depends on.

### Demand forecasting per zone

**Priority:** Stretch ✅ — **🔁 Milestone placement now confirmed**

**Resolution of open question #2:** [[1.rydex-product-overview|feature roadmap]] §7 placed this at Release Candidate by inference, flagging it for confirmation here. **Confirmed: Release Candidate is correct, but for a different reason than originally inferred.** The original inference tied it to the "evidence" theme of that milestone. The dependency-driven reason is more direct: forecasting needs a meaningful history of trip-request volume per zone to predict from — and that history doesn't exist with real signal until Beta 2 (zones, surge) and Beta 3 (a full trip lifecycle running with real usage patterns) have already shipped and run for a period. Release Candidate is therefore the _earliest_ point at which forecasting has enough real data to be meaningfully accurate, not merely the thematically-fitting slot.

**Acceptance criteria:**

- **Given** sufficient historical trip-request volume per zone exists, **when** a forecast is generated, **then** it predicts near-term demand with accuracy that is measured and reported, not asserted (per [[1-PRD|PRD]] §10)
- **Given** a forecast is generated, **when** displayed to a Driver or Admin, **then** it is presented as directional guidance, never as a guaranteed prediction

**Blocked by:** Zone & pricing rule management (Epic I), a meaningful volume of historical Trip data (Epic B running in production). **Blocks:** None.

### Ops dashboard

**Priority:** Advanced ✅

**Acceptance criteria:**

- **Given** trips are actively flowing through the system, **when** an Ops Agent or Admin opens the dashboard, **then** they see live trip volume, matching performance, and system health without needing engineering-level dashboard literacy

**Blocked by:** Trip status tracking (Epic B), System health view (Epic I, for the health portion specifically). **Blocks:** None.

### Historical trip analytics

**Priority:** Advanced ✅

**Acceptance criteria:**

- **Given** a meaningful volume of completed/cancelled trips exists, **when** an Admin requests historical analytics, **then** they see trends in volume, pricing, and cancellation rate over a selectable time range

**Blocked by:** Trip history & receipts (Epic B), Cancellation handling (Epic B). **Blocks:** None.

## G10. Epic I — Admin Console

> [!quote] Blocks: Zone & pricing rule management blocks Dynamic (surge) pricing (Epic E) directly — surge cannot apply to a zone that hasn't been defined.

### Zone & pricing rule management

**Priority:** Advanced ✅

**Acceptance criteria:**

- **Given** an Admin defines a new geographic zone (as a polygon) and base pricing rule, **when** saved, **then** the zone becomes immediately usable by pricing and matching logic without requiring an engineering deploy
- **Given** a Rider's pickup coordinate, **when** checked, **then** the system correctly determines which zone (if any) it falls within using point-in-polygon logic

**Blocked by:** None — foundational to Epic E's surge pricing and Epic H's per-zone forecasting. **Blocks:** Dynamic (surge) pricing (Epic E), Demand forecasting per zone (Epic H).

### Driver management

**Priority:** MVP ✅

**Acceptance criteria:**

- **Given** a Driver has submitted onboarding documents, **when** an Admin reviews them, **then** the Admin can approve (making the Driver eligible to go online) or reject with a reason
- **Given** an active Driver violates platform policy, **when** an Admin suspends them, **then** they are immediately ineligible for new matches, though any in-progress trip is allowed to complete

**Blocked by:** None — but functionally paired with Driver onboarding (Epic A), as flagged in §2 above. **Blocks:** Driver onboarding's approval step (Epic A) — this is a mutual, same-milestone dependency, not a one-directional block.

### System health view

**Priority:** Advanced ✅

**Acceptance criteria:**

- **Given** the platform's engineering observability data exists (per [[2-Engineering-Domain-Roadmap|Engineering Domain Roadmap]] 4.3), **when** an Admin opens the system health view, **then** the same health signal is presented in product terms, without requiring familiarity with Grafana/Jaeger/Prometheus directly

**Blocked by:** The engineering observability buildout itself (per [[3-End-to-End-Delivery-Plan|Delivery Plan]] Phase 7.1) — this feature has no product-side blocker but is fully blocked by an engineering deliverable, worth naming explicitly since it's easy to assume "Advanced priority" means "ready whenever the team gets to it" rather than "ready only after a specific engineering milestone." **Blocks:** Ops dashboard (shares the health-signal portion of its data).

## G11. Open Items Surfaced by This Pass

Per this project's standing rule — flag gaps rather than resolve them silently:

1. **Sender proof-of-delivery has no feature row anywhere in PRD §6, and therefore no Epic to traceability-map.** [[1.rydex-product-overview|personas]] §3 already flagged this as a need with no corresponding feature. **This matrix confirms the gap is real, not a misreading** — there is no Epic A–I row for Sender-specific delivery confirmation. Recommended next action: either (a) add a feature row to a future PRD revision (likely Epic B or a new "Epic J — Delivery") or (b) explicitly document that Sender reuses Rider's "trip complete, rate" flow as-is with no delivery-specific proof artifact, as a deliberate scope decision.
2. **AI support assistant (Epic G) depends on platform policy documentation existing as ingestable content** — this prerequisite is assumed throughout the PRD's AI sections but isn't owned by any Epic or any role in [[1.rydex-product-overview|personas]]. Recommended next action: assign ownership (likely the Platform Admin persona, or a new content-ownership responsibility) before Beta 3 planning finalizes.
3. **Driver onboarding (Epic A) and Driver management (Epic I) are mutually dependent**, not cleanly one-directional — flagged in §2 and §10 above. This is fine for Alpha (both ship together per [[1.rydex-product-overview|feature roadmap]] §3) but should be noted in [[3.rydex-domain-and-architecture|bounded contexts]] (Phase 1.3) so the two aren't modeled as fully separate bounded contexts if their data actually needs to be transactionally consistent (e.g., an approval action touching both). -e

## Part H — SLO Definitions

> [!abstract] Purpose of this document Per [[3-End-to-End-Delivery-Plan|Delivery Plan]] Phase 0.4: [[1-PRD|PRD]] §11 says things like "match found within a few seconds." This file turns every one of those six NFRs into an actual number with a stated measurement method. This is the file [[1.rydex-product-overview|success metrics]] pointed to for "where the exact number lives" — that file said _what_ matters and _why_; this file says _how much_ and _how it's verified_.

## I1. How to Read This Register

Each risk carries:

- **Source** — where the risk was first identified (PRD §14, an engineering domain, or a prior Phase 0 doc's flagged open item)
- **Likelihood / Impact** — a simple High/Medium/Low pairing, not a formal quantitative model (proportionate to this project's scale, per [[1.rydex-product-overview|Vision & Strategy]] §1's single-region scope)
- **Owner** — the role accountable for watching and mitigating it, per [[1.rydex-product-overview|personas]] where applicable
- **Mitigation** — the concrete action that reduces likelihood or impact, not just an acknowledgment that the risk exists
- **Tied to** — which Epic, SLO, or prior document section this risk traces back to

> [!note] A risk register is not a list of bad outcomes — it's a list of _owned_ mitigations Per [[1.rydex-product-overview|Vision & Strategy]] Pillar 4 (Evidence over assertion), an entry without a concrete mitigation isn't finished — it's just a worry written down. Every row below has an action, not only a description.

## I3. Engineering-Discovered Risks

### R-05 — Redis as system of record for the fairness queue

**Source:** [[3-End-to-End-Delivery-Plan|Delivery Plan]] §0.5 (named explicitly as the worked example for this section) **Description:** The driver fairness queue (Epic D) lives in a Redis sorted set, per [[2-Engineering-Domain-Roadmap|Engineering Domain Roadmap]] Tier 2.3. Redis offers two persistence modes (RDB, AOF) with different durability guarantees. Per Tier 2.3's own framing of what Rydex can afford to lose: location pings — yes; **driver queue state — no.** If Redis is configured with the wrong persistence mode (or none), a crash can silently lose the fairness queue's state, meaning drivers who'd been waiting longest lose their earned priority with no record it ever existed. **Likelihood:** Medium — persistence-mode misconfiguration is an easy default-settings mistake, not an exotic failure. **Impact:** High — this directly undermines the Match acceptance rate / fairness commitment that is one of [[1.rydex-product-overview|Vision & Strategy]] Pillar 1's explicit promises to the Driver persona. A silent loss here wouldn't throw an error — it would just quietly make the system less fair without anyone noticing until a driver complains. **Owner:** Senior Engineer (Backend, infrastructure-focus) **Mitigation:** Explicitly configure Redis with AOF (or RDB with a short save interval, evaluated against actual write volume) for the fairness-queue keyspace specifically — this may differ from the persistence mode used for the location-ping keyspace, since the two have opposite loss tolerances. **Action item:** document the chosen persistence mode and the reasoning in an ADR (per [[3-End-to-End-Delivery-Plan|Delivery Plan]] §1.1) before Beta 2 implementation begins, not after. **Tied to:** [[2-Engineering-Domain-Roadmap|Engineering Domain Roadmap]] Tier 2.3; Epic D (Fairness-aware queueing)

### R-06 — Driver onboarding and Driver management are mutually dependent, not cleanly layered

**Source:** [Traceability Matrix (Part G above)](https://claude.ai/chat/931cc535-a10d-4882-adff-50ece26a8b63#part-g--requirements-traceability-matrix) §2, §10, §11 (item 3) **Description:** Driver onboarding (Epic A) and Driver management's approval step (Epic I) depend on each other in a way that doesn't fit a clean one-directional Epic dependency — an approval action touches both. If modeled as fully separate bounded contexts without care, this coupling could force an awkward cross-context transaction or, worse, an eventually-consistent approval state that briefly disagrees with itself. **Likelihood:** Medium — this kind of coupling is easy to miss during initial bounded-context design and expensive to retrofit once contexts are drawn. **Impact:** Medium — primarily a design-quality risk (harder-to-maintain code, awkward transactions) rather than a user-facing failure, _if_ caught early. **Owner:** Senior Architect **Mitigation:** Explicitly resolve this when [[3.rydex-domain-and-architecture|bounded contexts]] (Phase 1.3) is written — either model Driver onboarding and Driver management's approval step as part of the same bounded context, or explicitly design the cross-context consistency mechanism (e.g., a domain event + eventual consistency, accepted deliberately) rather than discovering the coupling mid-implementation. **Tied to:** [[3.rydex-domain-and-architecture|domain glossary]] §9 (item 1 — also flags this); Epic A, Epic I

### R-07 — "Matched" status ambiguity in team communication

**Source:** [[3.rydex-domain-and-architecture|domain glossary]] §4, §9 (item 3) **Description:** The Trip status `Matched` means "an offer is outstanding," not "the driver has confirmed" — but this is exactly the kind of distinction that erodes in casual conversation, PR descriptions, and even code comments over time, especially as team size grows or memory of this glossary fades. **Likelihood:** Medium — language drift is a slow, easy-to-miss risk, not a single failure event. **Impact:** Low-Medium — the actual code enforces the correct state machine regardless of how people talk about it, so the practical risk is mainly in code _comments_, commit messages, and bug triage conversations leading reviewers astray, not in the system behaving incorrectly. **Owner:** Senior Architect (as glossary owner) + whole team (as practice) **Mitigation:** Link [[3.rydex-domain-and-architecture|domain glossary]] §4 in the project's contribution guidelines / PR template as required reading; treat "the trip is matched" used to mean "confirmed" as a flaggable terminology issue in code review, the same way a wrong variable name would be flagged. **Tied to:** [[3.rydex-domain-and-architecture|domain glossary]] §4

### R-08 — Real-time latency SLO conflates two different reliability guarantees

**Source:** [SLO Definitions (Part H above)](https://claude.ai/chat/931cc535-a10d-4882-adff-50ece26a8b63#part-h--slo-definitions) §3, §9 (item 1) **Description:** The current Real-time update latency SLO (p95 < 500ms) applies one latency number to both location pings (lossy-acceptable) and status pushes (must-not-lose). **Source:** [[3.rydex-domain-and-architecture|domain glossary]] §8, §9 (item 1) **Description:** Driver _identity_ clearly belongs to the Identity bounded context; Driver _presence_ (online/offline/busy) is operationally closest to the Matching context, since that's who consumes it. No explicit decision has been made about which context owns the presence attribute. **Likelihood:** Medium — same category of risk as R-06: easy to default into an inconsistent answer if not decided deliberately during bounded-context design. **Impact:** Medium — primarily a maintainability/design-clarity risk; a wrong-but-functional default is more likely than an outright failure, but it would make the codebase harder to reason about long-term. **Owner:** Senior Architect **Mitigation:** Resolve explicitly in [[3.rydex-domain-and-architecture|bounded contexts]] (Phase 1.3) as a named decision with stated reasoning, not a default. Given presence is Redis-backed, fast-changing, and primarily consumed by Matching (per [[3.rydex-domain-and-architecture|domain glossary]] §5), the likely correct answer is that Matching owns presence as a read model, while Identity remains the source of truth for the Driver record itself — but this should be a stated ADR-level decision, not an assumption. **Tied to:** [[3.rydex-domain-and-architecture|domain glossary]] §8; [[3-End-to-End-Delivery-Plan|Delivery Plan]] §1.3

### R-13 — Scalability SLO's placeholder number could be mistaken for an achieved result

**Source:** [SLO Definitions (Part H above)](https://claude.ai/chat/931cc535-a10d-4882-adff-50ece26a8b63#part-h--slo-definitions) §5, §9 (item 3) **Description:** The "500 concurrent active trips" figure in the Scalability SLO is explicitly a planning placeholder for designing the load test, not a committed or achieved number. There's a real risk that this figure gets cited later (in a demo script, a pitch deck, or casual conversation) as if it were an already-proven result. **Likelihood:** Medium — placeholder numbers are notoriously easy to "harden" into perceived facts over a long project, especially once they've been written down in one official-looking document. **Impact:** Medium — if cited externally as an achieved result before the Release Candidate load test actually runs, this would directly contradict [[1.rydex-product-overview|Vision & Strategy]] Pillar 4's "evidence over assertion" commitment, undermining exactly the credibility this whole project is designed to build. **Owner:** Senior PO / Business Analyst **Mitigation:** Before [[3-End-to-End-Delivery-Plan|Delivery Plan]] Phase 8 (Showcase Release) demo materials are finalized, explicitly audit all external-facing content for any scalability claim and confirm it cites the _actual_ measured ceiling from the Release Candidate's load-test report, not the "500" planning placeholder. **Tied to:** [SLO Definitions (Part H above)](https://claude.ai/chat/931cc535-a10d-4882-adff-50ece26a8b63#part-h--slo-definitions) §5
