# Rydex — Practical Engineering Execution Guide

> [!abstract] What this file is, and what it deliberately is not This file is the **literal, ordered sequence of engineering steps** for turning the four design documents into a real repository and a running system — from an empty repo through every milestone (Alpha, Beta 1, Beta 2, Beta 3, Release Candidate, Showcase Release) to production-credible evidence. It names every project, folder, file, and class that needs to exist, and explains the reasoning behind each sequencing decision.
> 
> **This file contains no implementation code.** Every step names _what_ to build and _why it goes here_, not _how to write it_ — the actual C#, SQL, and configuration logic is the engineer's work, not this document's. Where a class or pattern is named (e.g., `ValidationBehavior`), treat that as a naming and placement decision this guide is making, not a code sample to copy.
> 
> **Companion files in this set:**
> 
> - [[Rydex-Product-Overview]] — vision, strategy, personas, roadmap, success metrics
> - [[Rydex-Requirements-and-Risk]] — acceptance criteria, SLOs, and the risk register
> - [[Rydex-Domain-and-Architecture]] — domain model, ADRs, C4 diagrams, schema, matching engine design (the source of truth every step below implements)
> - [[Rydex-API-Contract]] — the OpenAPI v1 contract this guide's endpoints will eventually satisfy

---

## How to Read This Guide

Each step below has four parts:

- **What to do** — the concrete action (a command to run, a file to create, a folder to add)
- **Why it goes here, specifically** — the dependency reasoning that justifies this exact position in the sequence, not just "best practice"
- **What it depends on** — which earlier step must already be done
- **What it unlocks** — which later step becomes possible only after this one

> [!important] This guide assumes the four companion docs have already been validated Every entity name, value object, ADR decision, and acceptance criterion referenced below was already decided in [[Rydex-Domain-and-Architecture]] and [[Rydex-Requirements-and-Risk]]. This guide does not re-decide anything — it only sequences the _execution_ of decisions already made. If something named here doesn't match those files, the docs are the source of truth; flag the mismatch rather than trusting this guide's naming over them.

## Stage 1 — Repository & Solution Skeleton

### 1.1 Create the repository

**What to do:** Initialize a new Git repository named for the project (e.g. `rydex`). Add a `.gitignore` appropriate for .NET (covers `bin/`, `obj/`, `.vs/`, user-specific files). Add a root `README.md` stub — even a one-line placeholder — and a `LICENSE` file if applicable.

**Why it goes here, specifically:** Nothing else in this guide can exist without a place to live. This is also the moment to decide branch protection rules conceptually (even if not yet enabled in GitHub, since that's Stage 9.3) — per [[Rydex-Requirements-and-Risk|the Delivery Plan's cross-cutting practices]], ADRs and conventions should exist from commit one, not be retrofitted.

**Depends on:** Nothing.

**Unlocks:** Everything.

---

### 1.2 Create the solution

**What to do:** Create a single `.sln` file at the repository root (e.g. `Rydex.sln`). Do not create any projects yet — this step is just the solution container.

**Why it goes here, specifically:** Per [[Rydex-Domain-and-Architecture|the repository structure already specified]] (Delivery Plan §2.1), Rydex is a single solution containing multiple projects — this is the modular-monolith structure ADR-001 commits to for Alpha/Beta 1. One solution file is what lets all the projects below reference each other and build together.

**Depends on:** 1.1.

**Unlocks:** 1.3, 1.4.

---

### 1.3 Create the Web API project — show the architecture first

**What to do:** Create the **first** project as an ASP.NET Core Web API project, named `Rydex.Api`, and add it to the solution. At this point it should be the default template output — no custom folders yet, no real endpoints beyond whatever the template scaffolds.

**Why it goes here, specifically:** This matches your own instinct exactly — standing up the runnable shell first means there's something to `dotnet run` from day one, even before any real architecture exists inside it. It also gives the team (or future you) an immediate, visible "this is a real running thing" artifact, which matters psychologically on a long project as much as technically.

**Depends on:** 1.2.

**Unlocks:** 1.5 (once the library projects exist, `Rydex.Api` will reference them).

---

### 1.4 Create the class library projects — the architecture itself

**What to do:** Create the remaining projects as class libraries, one at a time, in this order, each added to the solution:

1. `Rydex.Domain`
2. `Rydex.Application`
3. `Rydex.Infrastructure`
4. `Rydex.Worker`
5. `Rydex.MatchingEngine` _(create the project shell now, per [[Rydex-Domain-and-Architecture|ADR-001]], but leave it essentially empty until Beta 2 — see the note below)_

Place all of them under `/src` at the repository root, matching the structure already named in [[Rydex-Domain-and-Architecture|the Delivery Plan §2.1]].

**Why this order, specifically:** `Rydex.Domain` first because every other project depends on it and it depends on nothing — it has zero framework references by design. `Rydex.Application` second because it depends only on `Rydex.Domain`. `Rydex.Infrastructure` third because it implements interfaces defined in `Rydex.Application` (repositories, etc.) and needs both lower layers to exist first. `Rydex.Worker` fourth because background services depend on Infrastructure being wireable. `Rydex.MatchingEngine` last, and intentionally near-empty, because per ADR-001 it isn't extracted as a real separate process until Beta 2 — creating the project shell now (rather than waiting) means the solution structure visually matches the target architecture from day one, but no one should be putting real logic in it yet.

**Depends on:** 1.2.

**Unlocks:** 1.5.

> [!note] Why create an empty MatchingEngine project this early at all This is a deliberate choice this guide is making, not something stated explicitly in the ADRs: showing the eventual extraction point in the solution structure from day one makes the Beta 2 extraction a "move code into an already-existing, already-referenced project" operation instead of a "stop and restructure the solution" operation later. The cost is one mostly-empty project sitting in the solution for a few months — a smaller cost than the alternative.

---

### 1.5 Wire up project references

**What to do:** Set project-to-project references following Clean Architecture's dependency rule:

- `Rydex.Application` → references `Rydex.Domain`
- `Rydex.Infrastructure` → references `Rydex.Domain` and `Rydex.Application`
- `Rydex.Worker` → references `Rydex.Application` and `Rydex.Infrastructure`
- `Rydex.Api` → references `Rydex.Application` and `Rydex.Infrastructure` (for DI registration only — never reference Infrastructure's concrete types directly in controllers)
- `Rydex.Domain` → references nothing else in the solution

**Why this order, specifically:** This is the literal enforcement of [[Rydex-Domain-and-Architecture|the Clean Architecture dependency rule]] named in the Domain & Architecture doc's component diagram notes — domain has no outward dependencies, and every other layer depends inward toward it, never the reverse. Getting this wrong now (e.g., accidentally letting `Rydex.Domain` reference `Rydex.Infrastructure`) is exactly the kind of mistake that's cheap to prevent here and expensive to unwind later.

**Depends on:** 1.3, 1.4.

**Unlocks:** Stage 2.

## Stage 2 — Generic / Cross-Cutting Scaffolding

> [!important] This is "the files every production platform like this has" This stage is exactly what you described — the common, almost boilerplate-feeling infrastructure that exists in essentially every serious ride-hailing/dispatch/marketplace backend, regardless of company. None of it is Rydex-specific yet; it's the scaffolding that makes the Rydex-specific code (Stage 3 onward) safe and consistent to write.

### 2.1 Solution-wide convention files

**What to do:** At the repository root, add:

- `.editorconfig` — code style rules (indentation, naming conventions, file-scoped namespaces or not)
- `Directory.Build.props` — shared MSBuild properties across all projects (nullable reference types enabled, implicit usings, language version, treat-warnings-as-errors policy)
- `global.json` (optional) — pins the .NET SDK version for consistent builds across machines

**Why it goes here, specifically:** These files apply to every project created in Stage 1 — adding them now means every project inherits the same conventions from its first build, rather than drifting and needing a later cleanup pass. This is the smallest, cheapest step in the entire guide, and skipping it is the single easiest thing to regret on a multi-month project.

**Depends on:** 1.5 (projects must exist for the props file to apply to anything).

**Unlocks:** Consistent style across everything that follows.

---

### 2.2 Domain layer's internal folders

**What to do:** Inside `Rydex.Domain`, create empty folders (no files yet) for:

- `Aggregates/` (or `Entities/`, depending on naming preference — pick one and use it consistently)
- `ValueObjects/`
- `Events/`
- `Enums/`
- `Common/` — for shared base types like an `AggregateRoot` base class or `IDomainEvent` marker interface, named here but not yet implemented (per your instruction, naming only)
- `Exceptions/` — for domain-specific exception types (e.g., an `InvalidTripStateTransitionException`, named per [[Rydex-Domain-and-Architecture|the Trip aggregate's invariant enforcement]])

**Why it goes here, specifically:** Per [[Rydex-Domain-and-Architecture|the vertical-slice and Clean Architecture guidance]], the Domain project's internal shape should be organized by _kind of domain concept_, not by feature — this is the one project in the solution that's deliberately NOT vertical-sliced, because it's small, foundational, and shared by every feature equally.

**Depends on:** 1.4 (Rydex.Domain project must exist).

**Unlocks:** Stage 3 (the actual domain types get placed into these folders).

---

### 2.3 Application layer's internal folders

**What to do:** Inside `Rydex.Application`, create:

- `Features/` — the root of the vertical-slice structure; subfolders are created per-feature as they're built (e.g., `Features/Trips/RequestTrip/`, `Features/Trips/CancelTrip/`, `Features/Drivers/OnboardDriver/`) — **do not pre-create every feature folder now; create each one only when that feature is actually being built**, per Stage 8's guidance
- `Common/` — for cross-cutting Application-layer types: the `IPipelineBehavior` implementations (named in 2.4 below), a `Result<T>` type (per [[Rydex-Domain-and-Architecture|the Result pattern]] named in the Engineering Roadmap), and shared interfaces like `ITripRepository`, `IDriverRepository` (interfaces only — implementations live in Infrastructure)
- `Interfaces/` (optional alternative to nesting interfaces inside `Common/` — pick one convention)

**Why it goes here, specifically:** Per [[Rydex-Domain-and-Architecture|the vertical-slice architecture]], `Features/` is intentionally empty at this stage — pre-building empty feature folders for features you haven't started yet (Matching, Pricing, etc.) just creates clutter and false signal about progress. `Common/` exists now because the pipeline behaviors and repository interfaces need to exist _before_ the first real feature slice is built in Stage 8, since every feature will depend on them.

**Depends on:** 2.2 (Application depends on Domain).

**Unlocks:** 2.4, Stage 8.

---

### 2.4 MediatR pipeline behaviors — named, not implemented

**What to do:** Inside `Rydex.Application/Common/Behaviors/` (or equivalent), create empty class files for each cross-cutting behavior named in [[Rydex-Domain-and-Architecture|the Engineering Roadmap's MediatR pipeline guidance]]:

- `ValidationBehavior.cs` — runs FluentValidation validators against every command/query before the handler executes
- `LoggingBehavior.cs` — structured logging of every request entering and leaving the pipeline
- `AuthorizationBehavior.cs` — checks resource-based/policy-based authorization before a handler runs, so individual handlers don't each need to re-implement auth checks
- `PerformanceBehavior.cs` — measures handler execution time, useful later for catching the kind of latency regressions the Time-to-match SLO cares about

**Why these four, specifically, and why now:** This is the literal list named in [[Rydex-Domain-and-Architecture|the Engineering Roadmap Tier 4.6]] — validation, logging, authorization, performance monitoring. They're created now, as empty named files, because every single feature slice built in Stage 8 onward will be registered into this same pipeline. Deciding the names and registration order now means no feature slice ever has to "remember" to add logging or validation — it's structurally guaranteed by the pipeline, not by developer discipline per-feature.

**Depends on:** 2.3.

**Unlocks:** Stage 8 (every feature handler will pass through this pipeline).

> [!note] Why "named, not implemented" matters most here This is the single best example in the entire guide of your instruction in practice: deciding that `ValidationBehavior` exists, what it's called, and where it sits in the pipeline order (validation → authorization → logging → performance, or whatever order is chosen) is an architectural decision worth documenting now. The actual FluentValidation wiring inside it is implementation detail that can be written once the pattern is agreed — and is explicitly left to you.

---

### 2.5 Infrastructure layer's internal folders

**What to do:** Inside `Rydex.Infrastructure`, create:

- `Persistence/` — will hold the DbContext (Stage 5.1), EF Core configuration classes (Stage 4), and migrations folder
- `Persistence/Configurations/` — one Fluent API configuration class per entity (populated in Stage 4)
- `Persistence/Migrations/` — EF Core will generate this automatically on first migration (Stage 6); no need to create it manually
- `Persistence/Dapper/` — Dapper query classes (Stage 5.3)
- `Repositories/` — concrete repository implementations (`TripRepository`, `DriverRepository`) implementing the interfaces named in 2.3
- `Messaging/` — RabbitMQ/MassTransit configuration and consumer registration (per [[Rydex-Domain-and-Architecture|ADR-004]])
- `Caching/` — Redis client wiring (per [[Rydex-Domain-and-Architecture|ADR-003]])
- `Identity/` — JWT issuance, token generation helpers (Stage 7)

**Why it goes here, specifically:** This is the project where every concrete technology choice from the ADRs actually lands — EF Core and Dapper (ADR-002), Redis (ADR-003), RabbitMQ (ADR-004). Organizing by _technology concern_ rather than by feature is correct specifically for this layer, since Infrastructure's job is implementing interfaces the Application layer defined, not expressing business logic.

**Depends on:** 2.2, 2.3 (Infrastructure depends on both Domain and Application).

**Unlocks:** Stage 4, Stage 5, Stage 7.

---

### 2.6 API layer's internal folders

**What to do:** Inside `Rydex.Api`, create:

- `Controllers/` (or organize as Minimal API endpoint groups, per [[Rydex-Domain-and-Architecture|ADR-005's REST decision]] — pick one consistently)
- `Middleware/` — custom middleware (correlation ID injection, exception-to-Problem-Details translation per [[Rydex-API-Contract|the RFC 7807 contract]])
- `Hubs/` — SignalR hub classes (created when Stage 9/Beta 1 real-time work begins; can stay empty for now)
- `Extensions/` — `IServiceCollection` extension methods for organizing DI registration cleanly (e.g., `AddInfrastructureServices()`, `AddApplicationServices()`) rather than one enormous `Program.cs`

**Why it goes here, specifically:** Keeping `Program.cs` thin by delegating to named extension methods is a small convention that pays off enormously as the number of registered services grows — by the time Beta 2/3 add Matching, Fraud, and AI Support registrations, a flat `Program.cs` would already be unreadable without this structure in place from the start.

**Depends on:** 1.3, 2.5 (Api references Infrastructure for DI wiring).

**Unlocks:** Stage 7.5 (auth endpoints), Stage 8 (the first real controller/endpoint).

---

### 2.7 Shared/common project — a decision point, not an automatic step

**What to do:** Decide whether Rydex needs a `Rydex.SharedKernel` (or `Rydex.Common`) project for types genuinely shared across the eventual Matching Engine process boundary (Beta 2+) — for example, the `Coordinate` value object, if the extracted Matching Engine process needs its own copy without referencing the full Domain project.

**Why this is flagged as a decision, not a default action:** At Alpha/Beta 1, everything lives in one process (per ADR-001), so `Rydex.Domain` alone is sufficient — there's no cross-process boundary yet requiring a shared contract project. Creating a `SharedKernel` project now, before it's needed, risks the same kind of premature complexity ADR-001 and ADR-005 both explicitly avoided elsewhere in this project. **Recommended default: skip this for now; revisit explicitly at the Beta 2 Matching Engine extraction (per [[Rydex-Domain-and-Architecture|ADR-001]]), not before.**

**Depends on:** Nothing yet — this is a deferred decision.

**Unlocks:** Nothing yet — intentionally.

## Stage 3 — Code-First Domain Model

> [!important] Why value objects come before aggregates This ordering is deliberate, not arbitrary. An aggregate's properties are _typed using_ value objects — you cannot correctly write `Trip.PickupLocation` as a `Coordinate` if `Coordinate` doesn't exist yet. Building bottom-up (smallest, most depended-upon types first) avoids placeholder types and rework.

### 3.1 Value objects first

**What to do:** Inside `Rydex.Domain/ValueObjects/`, create one file per value object named in [[Rydex-Domain-and-Architecture|Aggregates §K5 and the Domain Glossary §J6]]:

- `TripId.cs`
- `DriverId.cs`
- `RiderId.cs` _(per the open item flagged in [[Rydex-Domain-and-Architecture|Aggregates §K5]] — this was missing from the original brief and added as a correction; create it alongside the other two ID types now that the gap is confirmed)_
- `Coordinate.cs`
- `FareAmount.cs`

**Why this order, specifically:** ID value objects first (they have no dependencies on anything else), then `Coordinate` and `FareAmount` (also dependency-free, but logically "bigger" concepts than a bare ID). All five can technically be created in any order relative to each other — what matters is that all five exist _before_ Stage 3.2 begins, since every aggregate created there will reference them immediately.

**Depends on:** 2.2.

**Unlocks:** 3.2.

---

### 3.2 Aggregates and entities

**What to do:** Inside `Rydex.Domain/Aggregates/`, create one file per aggregate root named in [[Rydex-Domain-and-Architecture|Aggregates §K1]]:

- `Trip.cs`
- `Driver.cs`
- `Rider.cs`

Within each file, the properties should be typed using the value objects from 3.1 (e.g., `Trip.Id` is a `TripId`, not a raw `Guid`; `Trip.Pickup` is a `Coordinate`).

**Why this order, specifically:** Trip, Driver, and Rider have no dependency relationship _between_ each other at the domain layer — per [[Rydex-Domain-and-Architecture|Aggregates §K2]], `Trip` references `Driver` and `Rider` only by ID, never by direct object reference. This means the three aggregate files can genuinely be created in parallel or any order — there's no forced sequence among them, unlike the value-object-before-aggregate ordering above, which is a hard dependency.

**Depends on:** 3.1.

**Unlocks:** 3.3, 4.1.

> [!note] This is where the invariants get enforced, per the Domain & Architecture doc Per [[Rydex-Domain-and-Architecture|Aggregates §K2's invariant list]], the `Trip` aggregate's methods (not its public setters) are what should enforce: legal state transitions only, no `Matched` status without an assigned `DriverId`, immutability once terminal, and a required cancellation reason. This guide names that these methods need to exist (e.g., a `RequestTrip()` factory method, a `TransitionTo()` method, a `Cancel(reason)` method, consistent with the shape already sketched in [[Rydex-Domain-and-Architecture|Aggregates §K7's diagram]]) — writing their internal logic is implementation work left to you.

---

### 3.3 Domain events

**What to do:** Inside `Rydex.Domain/Events/`, create one file per domain event, using [[Rydex-Domain-and-Architecture|Domain Glossary §J7's five-event list]] as the authoritative source (confirmed there as more complete than the Delivery Plan's four-event brief):

- `TripRequestedEvent.cs`
- `TripMatchedEvent.cs`
- `TripStatusChangedEvent.cs`
- `TripCompletedEvent.cs`
- `DriverProfileUpdatedEvent.cs`
- `RiderProfileUpdatedEvent.cs` _(per the open item flagged in [[Rydex-Domain-and-Architecture|Aggregates §K8]] — this was identified as a likely gap; create it now alongside `DriverProfileUpdatedEvent` for symmetry, and flag it for product confirmation per that file's own recommendation)_

**Why it goes here, specifically:** Domain events are raised _by_ aggregates (per [[Rydex-Domain-and-Architecture|Aggregates §K2 and §K3]]'s "Domain events raised" sections) — they need to exist before the aggregate methods that raise them can compile, which is why this step follows 3.2 rather than preceding it.

**Depends on:** 3.2.

**Unlocks:** Stage 8 (events get consumed by the Outbox pattern once a real feature slice exists).

---

### 3.4 Enums and the TripStatus state machine type

**What to do:** Inside `Rydex.Domain/Enums/`, create:

- `TripStatus.cs` — the enum with values `Requested, Matched, DriverEnRoute, InProgress, Completed, Cancelled`, exactly matching [[Rydex-Domain-and-Architecture|Domain Glossary §J3's state machine]]
- `OnboardingStatus.cs` — `Pending, Approved, Rejected`
- `SuspensionStatus.cs` — per the open item in [[Rydex-Domain-and-Architecture|Schema v1 §Q7]], decide here whether this is a separate enum from a general "deactivated" concept, or the same one — this guide flags the decision point but does not resolve it, consistent with that file's own open item

**Why it goes here, specifically:** Enums are referenced by the aggregates created in 3.2, but listing them last in Stage 3 (rather than first) is intentional — by the time you're declaring `TripStatus`, you already know exactly which aggregate properties need it, which tends to produce a more accurate enum than designing it in isolation first.

**Depends on:** 3.2 (conceptually; can be created in parallel with 3.2 and 3.3 in practice).

**Unlocks:** Stage 4 (EF Core enum-to-string/int conversion configuration needs these to exist).

## Stage 4 — Relationships & EF Core Configuration

### 4.1 Fluent API configuration classes

**What to do:** Inside `Rydex.Infrastructure/Persistence/Configurations/`, create one configuration class per entity, implementing `IEntityTypeConfiguration<T>`:

- `TripConfiguration.cs`
- `DriverConfiguration.cs`
- `RiderConfiguration.cs`
- `FareConfiguration.cs`
- `TripEventConfiguration.cs`

**Why it goes here, specifically:** Per [[Rydex-Domain-and-Architecture|the Engineering Roadmap Tier 2.2]], Fluent API configuration is kept in dedicated classes (not inline in the DbContext) specifically so each entity's mapping concerns are isolated and reviewable on their own — this also matches the one-file-per-concept pattern used everywhere else in this guide.

**Depends on:** 3.2 (entities must exist to configure), 2.5 (folder must exist).

**Unlocks:** 4.2, 4.3, 5.1.

---

### 4.2 Owned-entity configuration for value objects

**What to do:** Inside the relevant configuration classes from 4.1 (primarily `TripConfiguration.cs`), name the configuration calls needed to map `Coordinate` (pickup and destination) and `FareAmount` as owned entities — per [[Rydex-Domain-and-Architecture|the Engineering Roadmap Tier 2.2]]'s explicit mention of "Fluent API for configuring value objects as owned entities (`Coordinate` embedded in `Trip`)."

**Why it goes here, specifically:** Value objects don't get their own top-level database table — they're embedded into their owning entity's table as columns. This needs explicit configuration because EF Core's default conventions don't know that `Coordinate` should be treated this way rather than as its own entity with its own table.

**Depends on:** 4.1, 3.1.

**Unlocks:** 4.3.

---

### 4.3 Relationship configuration

**What to do:** In the configuration classes, name the foreign-key relationships per [[Rydex-Domain-and-Architecture|Schema v1 §Q1's ERD]]:

- `Trip.RiderId` → `Rider` (required relationship)
- `Trip.DriverId` → `Driver` (optional/nullable relationship, since `DriverId` is null until `Matched`, per [[Rydex-Domain-and-Architecture|Schema v1 §Q2]])
- `Fare.TripId` → `Trip` (one-to-one)
- `TripEvent.TripId` → `Trip` (one-to-many, append-only)

**Why it goes here, specifically:** The nullable `Trip.DriverId` relationship is the one most likely to be configured incorrectly by default conventions — it needs to be explicitly marked optional, otherwise EF Core may infer a required relationship and the schema will reject perfectly valid `Requested`-status trips that have no driver yet.

**Depends on:** 4.1.

**Unlocks:** 4.4, 5.1.

---

### 4.4 Global query filters and conventions

**What to do:** Decide and name (in the relevant configuration classes or in the DbContext's `OnModelCreating`) any global query filters needed — for example, whether a soft-deleted/deactivated `Driver` should be filtered out of default queries automatically. Reference the open item in [[Rydex-Domain-and-Architecture|Schema v1 §Q7]] about whether deactivation needs its own dedicated column, and resolve it here at configuration time if it hasn't been resolved earlier.

**Why it goes here, specifically:** Global query filters are the mechanism that makes "drivers aren't deleted, they're deactivated" (per [[Rydex-Domain-and-Architecture|the Engineering Roadmap Tier 2.1]]) actually transparent to the rest of the codebase — without this, every single query against `Driver` would need to remember to manually exclude deactivated rows.

**Depends on:** 4.3.

**Unlocks:** Stage 5.

## Stage 5 — Data Access Setup

### 5.1 DbContext

**What to do:** Inside `Rydex.Infrastructure/Persistence/`, create `RydexDbContext.cs` — a single `DbContext` with one `DbSet<T>` per entity (`Trips`, `Drivers`, `Riders`, `Fares`, `TripEvents`, `OutboxMessages`, `ProcessedMessages`), and name the `OnModelCreating` override that applies all configuration classes from Stage 4.

**Why it goes here, specifically:** This is the single object that ties every Stage 4 configuration class together — it can't exist before the configurations do, but everything in Stage 5 onward (repositories, Dapper queries that share the same connection, migrations) depends on it existing.

**Depends on:** 4.1, 4.2, 4.3, 4.4.

**Unlocks:** 5.2, 5.4, Stage 6.

## Stage 6 — First Migration

### 6.1 Pre-migration checklist

**What to do:** Before generating anything, confirm:

- Every entity from [[Rydex-Domain-and-Architecture|Schema v1 §Q1]] has a corresponding `DbSet` in `RydexDbContext` (Stage 5.1)
- Every configuration class from Stage 4 is actually applied in `OnModelCreating`
- The connection string (Stage 5.2) points at a real, reachable PostgreSQL instance (even a local one)
- The `TripStatus` DB-level `CHECK` constraint design from [[Rydex-Domain-and-Architecture|Schema v1 §Q3]] has been decided — note that EF Core migrations don't generate complex `CHECK` constraints automatically in all cases, so this may require a manual addition to the generated migration file (see 6.3)

**Why it goes here, specifically:** A first migration generated against an incomplete model is the single most common source of migration-history mess on a new project — fixing a wrong first migration later (after a team has already applied it locally) creates exactly the kind of "many small migrations undoing each other" history that makes a schema hard to reason about. Checking once, carefully, before running the command is cheap insurance.

**Depends on:** Stage 4, Stage 5 fully complete.

**Unlocks:** 6.2.

---

### 6.2 Generate the migration

**What to do:** Run the EF Core migrations-add command (via `dotnet ef migrations add <Name>`, run from a context where the CLI tooling is installed and pointed at `Rydex.Infrastructure` as the project containing the DbContext) with a clear, descriptive migration name — e.g., `InitialCreate`.

**Why it goes here, specifically:** This is the first command in this entire guide that actually produces a database-shape artifact. Everything from Stage 1 through Stage 5 was preparing the model; this step is where that model becomes a concrete, versioned migration file that can be reviewed, committed, and applied.

**Depends on:** 6.1.

**Unlocks:** 6.3.

---

### 6.3 Review the generated migration

**What to do:** Open the generated migration file and manually verify, against [[Rydex-Domain-and-Architecture|Schema v1]]:

- The `Trip` table includes the `IdempotencyKey` unique constraint (per [[Rydex-Domain-and-Architecture|Schema v1 §Q2]])
- The three named indexes from [[Rydex-Domain-and-Architecture|Schema v1 §Q4]] are present — the composite `(DriverId, Status)` index, the partial index for active trips, and the covering index for trip-history pagination
- The `RowVersion` concurrency token columns exist on `Trip` and `Driver` (per [[Rydex-Domain-and-Architecture|Schema v1 §Q5]])
- The geography-typed columns and the GIST spatial index (per [[Rydex-Domain-and-Architecture|Schema v1 §Q4]]'s flagged addition) are present — these often require manual `migrationBuilder.Sql(...)` additions since EF Core's default provider may not generate PostGIS-specific syntax automatically
- The `TripStatus` state-machine `CHECK` constraint (per [[Rydex-Domain-and-Architecture|Schema v1 §Q3]]) is present — likely requiring a manual addition, as flagged in 6.1

**Why it goes here, specifically:** EF Core's automatic migration generation is reliable for standard relational shapes but is known to miss or simplify exactly the kind of advanced constraints (`CHECK` constraints encoding a state machine, PostGIS geography types, partial indexes) that [[Rydex-Domain-and-Architecture|Schema v1]] specifically calls for. This review step exists because trusting the generated file blindly here is the most likely place in the whole guide for the real database to silently diverge from the documented schema.

**Depends on:** 6.2.

**Unlocks:** 6.4.

### 6.4 Apply the migration

**What to do:** Run the EF Core database-update command against the local/dev database, then verify the resulting schema (via `psql` or a GUI client) actually matches what was reviewed in 6.3.

**Why it goes here, specifically:** Applying without a final visual check defeats the purpose of 6.3's review — this step closes the loop by confirming the _applied_ schema, not just the _generated migration file_, matches intent.

**Depends on:** 6.3.

**Unlocks:** Stage 7, Stage 8.

**Why it goes here, specifically:** Flagged as optional and last in this stage because seed data is a convenience for local development, not a structural requirement — it shouldn't block or complicate the first real migration, but it's worth deciding deliberately now rather than ad-hoc later once multiple developers are inserting test data inconsistently by hand.

**Depends on:** 6.4.

**Unlocks:** Nothing structural — purely a developer-experience convenience.

## Stage 7 — Identity, Roles, and Auth

### 7.1 Identity model decision

**What to do:** Decide explicitly: is Rydex using ASP.NET Core Identity (the framework's built-in user/role system) as the backing store for Rider/Driver authentication, or a fully custom identity implementation built directly against the `Rider`/`Driver` aggregates already designed in Stage 3?

**Why this is flagged as a decision, not a default:** Per [[Rydex-Domain-and-Architecture|Bounded Contexts §L2]], Identity is a clear bounded context, but neither the PRD nor the Engineering Roadmap explicitly commits to ASP.NET Core Identity by name — they describe the _behavior_ needed (JWT issuance, role claims, refresh token rotation) without mandating the specific framework mechanism. **Recommended consideration:** ASP.NET Core Identity brings a lot of built-in plumbing (password hashing, lockout policies) but also its own user/role table shape, which would need careful reconciliation with the `Rider`/`Driver` aggregates already designed with their own identity-level fields (per [[Rydex-Domain-and-Architecture|Aggregates §K3, §K4]]) — a fully custom approach avoids that reconciliation but means re-implementing some of what Identity provides for free. This guide does not make this decision for you; it flags that the decision must be made explicitly before 7.2 can proceed meaningfully.

## Stage 8 — First Vertical Slice (Proving the Architecture)

### 8.1 Why this stage exists

Everything from Stage 1 through Stage 7 is scaffolding and foundational types — none of it, on its own, proves the architecture actually works end-to-end. This stage is the first feature built completely, top to bottom, specifically to validate that every layer wired up so far actually connects correctly before building the rest of Rydex on top of an unproven foundation.

### 8.2 Pick the proving feature

**What to do:** Build **Request a Trip** first (per [[Rydex-Requirements-and-Risk|the traceability matrix §G3]]) — not because it's the simplest feature, but because it touches the most layers at once: Domain (the `Trip` aggregate's creation logic), Application (a `RequestTripCommand` and its handler, passing through every Stage 2.4 pipeline behavior), Infrastructure (the repository, the DbContext, the Idempotency-Key deduplication check), and Api (the actual `POST /trips` endpoint per [[Rydex-API-Contract|the contract]]).

**Why this feature, specifically:** Choosing the single most-connected feature as the proving slice means that if something is wrong in the scaffolding (a missing DI registration, a misconfigured pipeline behavior, a wrong connection string), it surfaces immediately on the first real feature — rather than three features later, once the mistake has been silently copy-pasted into other slices.

**Depends on:** All of Stages 1–7.

**Unlocks:** 8.3.

---

### 8.3 Build the full slice

**What to do:** Inside `Rydex.Application/Features/Trips/RequestTrip/` (created now, per Stage 2.3's "create per-feature, when needed" guidance), name:

- `RequestTripCommand.cs`
- `RequestTripCommandHandler.cs`
- `RequestTripCommandValidator.cs` (the FluentValidation validator that `ValidationBehavior`, named in Stage 2.4, will pick up automatically)
- `RequestTripResponse.cs` (or reuse a shared `TripDto`)

Then, in `Rydex.Api`, name the controller action (or minimal API endpoint) that accepts `POST /trips`, accepts the `Idempotency-Key` header per [[Rydex-API-Contract|the contract]], and dispatches `RequestTripCommand` via MediatR.

**Why this order, specifically:** Command → Handler → Validator → Endpoint mirrors the request's actual runtime flow backwards from where the logic lives to where the request enters — building it in this order means each piece can be reasoned about in isolation before the next one is added.

**Depends on:** 8.2.

**Unlocks:** 8.4, and every subsequent Alpha feature (per [[Rydex-Product-Overview|the Feature Roadmap's Alpha milestone]]) — Driver onboarding, Trip status tracking, Nearest-driver matching, and so on, each following this same slice pattern.

---

### 8.4 What "done" looks like here

A request hits `POST /trips`, passes through `ValidationBehavior` → `AuthorizationBehavior` → `LoggingBehavior` → `PerformanceBehavior` (or whatever final order is chosen in Stage 2.4), reaches `RequestTripCommandHandler`, creates a `Trip` aggregate via its domain method (not a raw constructor, per Stage 3.2's note), persists it via `ITripRepository` (Stage 5.4) through `RydexDbContext` (Stage 5.1) into the schema applied in Stage 6, and returns a `202 Accepted` per [[Rydex-API-Contract|the contract]]. If all of that happens correctly on the first real feature, the scaffolding from Stages 1–7 is validated — everything after this is building more features on a now-proven foundation, not discovering foundational problems mid-feature.

## Stage 12 — Beta 1: The Real-Time Layer

> [!quote] Proves: the product feels real-time — live position updates within target latency, status changes push without refresh, and killing one backend instance mid-trip doesn't drop messages.

### 12.1 SignalR hub architecture

**What to do:** Inside `Rydex.Api/Hubs/` (the folder named but left empty back in Stage 2.6), create `TripHub.cs` with a strongly-typed client interface, using `trip:{tripId}` groups so location and status updates go only to the rider and driver on that specific trip, not broadcast globally. **Why rules before ML, specifically:** Per [[Rydex-Domain-and-Architecture|the Delivery Plan §6.1's explicit sequencing]], deterministic rules are cheap, explainable, and catch the obvious cases — building them first means there's already a working, if simple, fraud signal in place before investing in the harder-to-validate ML layer.

### 14.3 RAG support assistant

**What to do:** Following [[Rydex-Domain-and-Architecture|the Engineering Roadmap Tier 3.5]]'s pipeline exactly: name the document-ingestion and chunking step (semantic chunking, chunk size tested empirically against retrieval quality, not guessed), the embedding step (via a hosted model into pgvector, per [[Rydex-Domain-and-Architecture|ADR-007]]), the query-time retrieval step (embed question → cosine similarity search → top-K chunks), and the synthesis step (an LLM call with a system prompt explicitly instructed to answer only from retrieved context and say "I don't know" otherwise).
