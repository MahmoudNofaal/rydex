## S2. Resource Modeling Decisions

**Resource-oriented, not action-oriented**, per the explicit instruction in [[2-Engineering-Domain-Roadmap|Engineering Domain Roadmap]] Tier 2.5: `POST /trips`, not `POST /requestTrip`. Sub-actions that don't fit a pure REST verb (`accept`, `cancel`, `approve`, `suspend`) are modeled as `POST` against a sub-resource path (`/trips/{tripId}/cancel`), which is the same pattern the Delivery Plan's own brief names explicitly (`/trips/{id}/cancel`).

**Status codes chosen deliberately, not defaulted:**

- `202 Accepted` on `POST /trips` — matching happens asynchronously; the response does not contain a match result, only the created Trip in `Requested` status (per [[2-Engineering-Domain-Roadmap|Engineering Domain Roadmap]] Tier 3.3's sync/async framing)
- `409 Conflict` on `acceptTrip` and `cancelTrip` — covers both the double-booking race (per [[3.rydex-domain-and-architecture|Schema v1]] §5's `RowVersion` concurrency) and illegal state transitions (per [[3.rydex-domain-and-architecture|Domain Glossary]] §3)
- `422 Unprocessable Entity` reserved specifically for semantic validation failures, distinct from `400`-style malformed-request errors

**HATEOAS-light `_links`** on the `Trip` schema — per [[2-Engineering-Domain-Roadmap|Engineering Domain Roadmap]] Tier 2.5, this tells a client which actions are currently valid (`cancel`, `rate`, `accept`) without the client needing to independently re-derive that from `status` — the server is the single source of truth for what's currently legal, consistent with the Trip aggregate being the only thing allowed to know its own valid transitions ([[3.rydex-domain-and-architecture|Aggregates]] §2).

## S3. What's Deliberately Not in This Version

- **No `/trips/{tripId}/sos` endpoint** — SOS/emergency flag is tagged Stretch in [[1.rydex-product-overview|feature roadmap]] §6 and belongs to Beta 3; omitted from this v1 contract rather than speculatively included.
- **No `/admin/zones` endpoints** — Zone & pricing rule management (Epic I) belongs to Beta 2 and depends on the still-undecided Admin Console container placement ([[3.rydex-domain-and-architecture|C4 Component Diagram]] §4's open item) — adding these endpoints now would mean designing against an architectural decision that hasn't been made yet.
- **No real-time/WebSocket contract** — SignalR hub contracts are a different artifact entirely (per [[2-Engineering-Domain-Roadmap|Engineering Domain Roadmap]] Tier 3.1) and don't belong in an OpenAPI/REST spec; this file covers only the synchronous REST surface.
- **No `/matching` endpoints** — per [[3.rydex-domain-and-architecture|Bounded Contexts]] §6, Matching is consumed internally by Trip Lifecycle (`requestTrip` triggers it asynchronously); it isn't a directly client-callable resource in this version.

## S5. Open Items Surfaced by This Pass

1. **No refresh-token endpoint is defined**, despite [[2-Engineering-Domain-Roadmap|Engineering Domain Roadmap]] Tier 2.4 naming "refresh token rotation" as a required Auth domain. `POST /auth/login` returns both tokens, but there's no `POST /auth/refresh` in this v1 contract — flagged as a gap to close before Alpha implementation, not deferred intentionally.
2. **Driver presence (online/offline) has no REST endpoint** — per [[3.rydex-domain-and-architecture|Bounded Contexts]] §4, presence is owned by Matching and is Redis-backed; it's plausible this should be a SignalR hub method (`GoOnline()`/`GoOffline()`) rather than a REST endpoint, consistent with presence being fast-changing, ephemeral state. **Recommended action:** confirm this is intentionally excluded from the REST contract (because it belongs in the real-time hub contract instead) rather than simply forgotten.
3. **Admin-facing endpoints are minimal** (`approve`/`suspend` only) — full Admin Console functionality (zone management, system health view) is intentionally deferred per §3 above, but this means the v1 contract is not yet a complete surface for the Platform Admin persona. -e

## Part T — Full OpenAPI v1 Specification (YAML)

```yaml
openapi: 3.0.3
info:
  title: Rydex API
  version: "1.0.0"
  description: >
    Rydex ride-hailing and delivery dispatch platform — public API.
    Resource-oriented, versioned from day one (/v1/), with Idempotency-Key
    support on trip creation, cursor pagination on trip history, and
    RFC 7807 Problem Details for all errors.
    See docs/3-End-to-End-Delivery-Plan.md §1.5 for the brief this spec fulfills.

servers:
  - url: /v1

tags:
  - name: Trips
  - name: Drivers
  - name: Riders
  - name: Auth

paths:
  /trips:
    post:
      tags: [Trips]
      summary: Request a trip
      description: >
        Creates a Trip in Requested status and begins matching asynchronously.
        Maps to RequestTripCommand (Engineering Roadmap Tier 1.3) and the
        Request a trip acceptance criteria in requirements/traceability-matrix.md §3.
      operationId: requestTrip
      parameters:
        - name: Idempotency-Key
          in: header
          required: true
          description: >
            Client-generated key. A retried request with the same key never
            creates a second Trip (traceability-matrix.md §3; enforced by a
            unique constraint on Trip.IdempotencyKey, schema-v1.md §6).
          schema:
            type: string
            format: uuid
      requestBody:
        required: true
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/RequestTripBody'
      responses:
        '202':
          description: >
            Trip created in Requested status; matching is happening
            asynchronously (Engineering Roadmap Tier 2.5 — 202 Accepted for
            async operations). Match result is pushed via the real-time
            channel, not returned in this response.
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/Trip'
        '422':
          $ref: '#/components/responses/ValidationProblem'
        '401':
          $ref: '#/components/responses/Unauthorized'
        default:
          $ref: '#/components/responses/Problem'

  /trips/history:
    get:
      tags: [Trips]
      summary: Get paginated trip history for the authenticated Rider or Driver
      description: >
        Cursor-based pagination — never offset, since offset breaks when new
        trips are inserted (Engineering Roadmap Tier 2.5). Backed by the
        Dapper read path (ADR-002) and the covering index defined in
        schema-v1.md §4.
      operationId: getTripHistory
      parameters:
        - name: cursor
          in: query
          required: false
          schema:
            type: string
          description: Opaque cursor from a previous response's nextCursor field.
        - name: limit
          in: query
          required: false
          schema:
            type: integer
            minimum: 1
            maximum: 100
            default: 20
      responses:
        '200':
          description: A page of trip history.
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/TripHistoryPage'
        '401':
          $ref: '#/components/responses/Unauthorized'
        default:
          $ref: '#/components/responses/Problem'

  /trips/{tripId}:
    get:
      tags: [Trips]
      summary: Get trip details
      operationId: getTripDetails
      parameters:
        - $ref: '#/components/parameters/TripId'
      responses:
        '200':
          description: Trip detail, including embedded Fare and HATEOAS-light _links.
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/Trip'
        '403':
          description: >
            Resource-based authorization failure — a Rider may only view
            their own trips (Engineering Roadmap Tier 2.4; resource-based,
            not just role-based, authorization).
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/Problem'
        '404':
          $ref: '#/components/responses/NotFound'
        default:
          $ref: '#/components/responses/Problem'

  /trips/{tripId}/cancel:
    post:
      tags: [Trips]
      summary: Cancel a trip
      description: >
        Always captures a cancellation reason (traceability-matrix.md §3).
        Rejected if the Trip is already in a terminal state
        (domain/aggregates.md §2 — terminal-state immutability invariant).
      operationId: cancelTrip
      parameters:
        - $ref: '#/components/parameters/TripId'
      requestBody:
        required: true
        content:
          application/json:
            schema:
              type: object
              required: [reason]
              properties:
                reason:
                  type: string
                  description: Required cancellation reason.
      responses:
        '200':
          description: Trip moved to Cancelled.
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/Trip'
        '409':
          description: >
            Trip is already in a terminal state — cancellation is not a
            legal transition from here (domain/glossary.md §3).
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/Problem'
        '422':
          $ref: '#/components/responses/ValidationProblem'
        default:
          $ref: '#/components/responses/Problem'

  /trips/{tripId}/accept:
    post:
      tags: [Trips]
      summary: Driver accepts an outstanding match offer
      description: >
        This is the action that transitions a Trip from Matched to
        DriverEnRoute (domain/glossary.md §4 — the precise Matched-vs-Accepted
        distinction). Protected against the double-booking race via
        optimistic concurrency on the Trip row (schema-v1.md §5).
      operationId: acceptTrip
      parameters:
        - $ref: '#/components/parameters/TripId'
      responses:
        '200':
          description: Trip transitioned to DriverEnRoute.
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/Trip'
        '409':
          description: >
            Concurrent acceptance race lost, or the offer is no longer valid
            (e.g. it already timed out and was re-offered to another Driver)
            — Engineering Roadmap Tier 2.5's named use of 409 for concurrent
            driver booking.
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/Problem'
        default:
          $ref: '#/components/responses/Problem'

  /trips/{tripId}/rating:
    post:
      tags: [Trips]
      summary: Submit a rating for a completed trip
      description: >
        Only permitted once Trip.Status = Completed
        (traceability-matrix.md §3 — Ratings & feedback).
      operationId: submitTripRating
      parameters:
        - $ref: '#/components/parameters/TripId'
      requestBody:
        required: true
        content:
          application/json:
            schema:
              type: object
              required: [score]
              properties:
                score:
                  type: integer
                  minimum: 1
                  maximum: 5
                comment:
                  type: string
      responses:
        '201':
          description: Rating recorded.
        '409':
          description: Trip is not yet Completed.
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/Problem'
        default:
          $ref: '#/components/responses/Problem'

  /drivers:
    post:
      tags: [Drivers]
      summary: Driver onboarding
      description: >
        Creates a Driver in pending-verification onboarding status
        (traceability-matrix.md §2). Not eligible to go online until an
        Admin approves (Epic I — resolved as the same Identity-context
        operation per domain/bounded-contexts.md §3).
      operationId: onboardDriver
      requestBody:
        required: true
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/DriverOnboardingBody'
      responses:
        '201':
          description: Driver created, OnboardingStatus = Pending.
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/Driver'
        '422':
          $ref: '#/components/responses/ValidationProblem'
        default:
          $ref: '#/components/responses/Problem'

  /drivers/{driverId}:
    get:
      tags: [Drivers]
      summary: Get driver profile
      operationId: getDriver
      parameters:
        - $ref: '#/components/parameters/DriverId'
      responses:
        '200':
          description: Driver profile.
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/Driver'
        '404':
          $ref: '#/components/responses/NotFound'
        default:
          $ref: '#/components/responses/Problem'

  /drivers/{driverId}/approve:
    post:
      tags: [Drivers]
      summary: Admin approves a driver's onboarding
      description: >
        Identity-context operation (domain/bounded-contexts.md §3 — resolves
        risk-register.md R-06). Restricted to the Admin role.
      operationId: approveDriver
      parameters:
        - $ref: '#/components/parameters/DriverId'
      responses:
        '200':
          description: Driver OnboardingStatus = Approved.
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/Driver'
        '403':
          $ref: '#/components/responses/Forbidden'
        default:
          $ref: '#/components/responses/Problem'

  /drivers/{driverId}/suspend:
    post:
      tags: [Drivers]
      summary: Admin suspends a driver
      description: >
        Driver becomes immediately ineligible for new matches; an
        in-progress trip is allowed to complete (traceability-matrix.md §10).
      operationId: suspendDriver
      parameters:
        - $ref: '#/components/parameters/DriverId'
      requestBody:
        required: true
        content:
          application/json:
            schema:
              type: object
              required: [reason]
              properties:
                reason:
                  type: string
      responses:
        '200':
          description: Driver SuspensionStatus = Suspended.
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/Driver'
        '403':
          $ref: '#/components/responses/Forbidden'
        default:
          $ref: '#/components/responses/Problem'

  /riders:
    post:
      tags: [Riders]
      summary: Rider signup
      operationId: signupRider
      requestBody:
        required: true
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/RiderSignupBody'
      responses:
        '201':
          description: Rider account created.
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/Rider'
        '422':
          $ref: '#/components/responses/ValidationProblem'
        default:
          $ref: '#/components/responses/Problem'

  /riders/{riderId}:
    get:
      tags: [Riders]
      summary: Get rider profile
      operationId: getRider
      parameters:
        - $ref: '#/components/parameters/RiderId'
      responses:
        '200':
          description: Rider profile.
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/Rider'
        '403':
          $ref: '#/components/responses/Forbidden'
        '404':
          $ref: '#/components/responses/NotFound'
        default:
          $ref: '#/components/responses/Problem'

  /auth/login:
    post:
      tags: [Auth]
      summary: Email/phone-based login
      operationId: login
      requestBody:
        required: true
        content:
          application/json:
            schema:
              type: object
              required: [identifier, password]
              properties:
                identifier:
                  type: string
                  description: Email or phone.
                password:
                  type: string
                  format: password
      responses:
        '200':
          description: Authenticated; returns access + refresh tokens.
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/AuthTokens'
        '401':
          $ref: '#/components/responses/Unauthorized'
        default:
          $ref: '#/components/responses/Problem'

components:
  parameters:
    TripId:
      name: tripId
      in: path
      required: true

  schemas:

          type: string
          format: uuid

        pickup:
          $ref: '#/components/schemas/Coordinate'
        destination:
          $ref: '#/components/schemas/Coordinate'
        fare:
          $ref: '#/components/schemas/Fare'
        requestedAt:
          type: string

        _links:
```
